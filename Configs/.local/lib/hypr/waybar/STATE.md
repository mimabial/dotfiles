# Waybar state contract

State is read by three different code paths in this subsystem, each with a
different reason for existing. They MUST agree on the file format. This
document is the canonical description of that format.

## File format

Two files, both plain `KEY=value` per line:

| File | Path | Owner | Format notes |
|---|---|---|---|
| **staterc** | `${XDG_STATE_HOME}/hypr/staterc` | `core/state.sh` (writers across the codebase) | `KEY=value` lines. May contain bash quoting (`KEY="value"` or `KEY='value'`); readers must strip surrounding quotes. May contain `export ` prefix; readers must strip it. |
| **env-overrides** | `${XDG_STATE_HOME}/hypr/env-overrides` | per-host generated by `runtime/init.bash` | Same format as staterc. Read after staterc; no precedence between them in the bash reader. |

Lock-meta files (a separate format used internally by waybar):

| File | Path | Format |
|---|---|---|
| **WAYBAR_WATCH_META** | `${XDG_RUNTIME_DIR}/hypr/waybar_watch_meta` | `KEY=value` lines, no quoting, no export prefix. Written by `waybar_watch.watch_waybar` with keys `pid`, `started`, `cmd`. |
| **THEME_UPDATE_META** | `${XDG_RUNTIME_DIR}/hypr/theme_update_meta` | Same as WAYBAR_WATCH_META. Written by `theme.apply.sh`; key `waybar_reload=css-hot` tells the watcher that theme changes are expected to be CSS-only and should be handled by Waybar's style hot reload. |

## Three readers, three reasons

### 1. Python — `waybar_state.get_state_value(key, default=None)`

Used by the management path (`waybar.py --update`, layout switches, the apply
pipeline). Reads `staterc` line-by-line, returns the first match. No quote
stripping — relies on staterc not containing quoted values for waybar keys.

**Why a separate reader:** the management path needs lock-coordinated
writes too (`set_state_value`, `set_state_values`, `state_write_lock`).
Same module owns both sides.

### 2. Python — `waybar_watch.read_runtime_meta(lock_path)`

Used by the watcher process to read both its own meta and the theme-update
meta. Format is stricter: no quoting, no `export`, no whitespace tolerance
beyond `key.strip() = value.strip()`. **Different file format** from
staterc (the meta files are written by Python with controlled content;
staterc is shell-written and may contain shell quoting).

**Why a separate reader:** the lock-meta files are a Python-Python channel;
shell quoting tolerance would just be dead code.

### 3. Bash — `waybar.state.common.sh:waybar_state_value(key, default)`

Used by indicator scripts that emit JSON for waybar custom modules
(currently only `waybar.colormode.sh`). Reads staterc and env-overrides
WITHOUT sourcing them (sourcing arbitrary user shell could execute code).
Strips `export ` prefix and surrounding `"`/`'` quotes.

**Why a separate reader:** Python startup is ~80 ms, too slow for a Waybar
module with `interval: 1`. The bash reader is the same canonical format,
implemented natively to avoid the Python tax. If `state_get` (from
`core/state.sh`) is in scope, it delegates there for consistency.

## Adding a new state key

1. Pick a name. By convention waybar keys are `WAYBAR_*` (e.g.
   `WAYBAR_LAYOUT_PATH`, `WAYBAR_STYLE_PATH`).
2. Use `waybar_state.set_state_value(key, value)` from Python; or
   `state_set "$key" "$value" "staterc"` from bash. Do not write directly.
3. Read via the appropriate path-specific reader above. Never source
   staterc into a shell — it can contain `export VAR=...` lines whose
   right-hand side is unsanitized.

## Format invariants the three readers agree on

- One assignment per line.
- Comments (`# ...`) and blank lines are ignored.
- Leading whitespace is tolerated.
- The Python staterc reader does **not** strip quotes; the bash reader
  **does**. Any caller writing a value that contains quotes must therefore
  decide which reader will consume it. In practice: keep values quote-free.
- Lock-meta files (the Python-Python channel) never contain quotes,
  comments, or `export`.
